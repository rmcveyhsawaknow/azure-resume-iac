# [Phase 5] Add Cloudflare rate-limiting rule to protect anonymous counter endpoint

## Description

The `GetResumeCounter` Azure Function uses `AuthorizationLevel.Anonymous` so that the
visitor counter works without embedding a function key in the browser-visible frontend.
`AuthorizationLevel.Function` was evaluated but caused HTTP 401/404 errors in browser
network traces because the frontend had no mechanism to supply the key at request time
without hard-coding it in source (itself a security anti-pattern).

The `Anonymous` level is therefore an intentional, documented choice. However it means
the counter endpoint is publicly callable with no authentication barrier, which creates
two risks:

1. **Abuse / scripted increments** — automated traffic can inflate the counter arbitrarily.
2. **Cost spikes** — unexpected call volumes drive up Azure Function execution and Cosmos DB
   RU consumption.

The accepted mitigation is to add a **Cloudflare rate-limiting rule** in front of the
Function App origin so that Cloudflare blocks or challenges excessive requests before they
reach Azure.

## Acceptance Criteria

- [ ] Cloudflare Rate Limiting rule created for the Function App custom domain path (e.g., `resume.ryanmcvey.me/api/*` for prod, `resumedev.ryanmcvey.me/api/*` for dev) so that Cloudflare blocks/challenges before traffic reaches the Azure Functions origin
- [ ] Rule threshold defined (e.g., ≤ 10 requests / 10 s per IP) and documented in `docs/KNOWN_ISSUES.md`
- [ ] Rule is managed via a Cloudflare API shell script (following the pattern of `scripts/cloudflare-dns-record.sh`) and integrated into the deployment workflow
- [ ] Tested: legitimate browser access (1–2 page loads) passes; simulated burst (> threshold) is challenged/blocked
- [ ] `docs/KNOWN_ISSUES.md` updated to reflect mitigation as **RESOLVED** once rule is deployed
- [ ] No function key hard-coded in frontend source

## Background / Context

- **Original issue:** `frontend/main.js` contained a hard-coded Azure Function URL with a
  function key query-string (`?code=…`). This was identified as a security anti-pattern in
  the Phase 0 assessment (see `docs/KNOWN_ISSUES.md` — "Hardcoded Secrets in Source Control").
- **Attempted fix:** Switching to `AuthorizationLevel.Function` and injecting the key via
  CI/CD was prototyped but resulted in HTTP 401 and 404 errors in the browser because the
  frontend fetch call did not include the key at runtime. The JavaScript would need a secure
  mechanism to receive and attach the key without exposing it in source, which adds
  significant complexity for a low-value, low-risk counter endpoint.
- **Decision:** `AuthorizationLevel.Anonymous` was chosen as the pragmatic option.
  Cloudflare rate limiting is the agreed compensating control.
- **References:** PR #223 review thread, `backend/api/GetResumeCounter.cs` line 20,
  `docs/KNOWN_ISSUES.md` section "Hardcoded Secrets in Source Control".

## Notes

`AuthorizationLevel.Anonymous` is not a bug — it is a deliberate architectural choice
with a documented compensating control. Do not change the auth level back to `Function`
without first implementing a secure key-injection mechanism for the frontend (e.g.,
injecting the key via `config.js` at deploy time and reading it from `window.FUNCTION_KEY`
in `main.js`). That approach is tracked separately and would allow the auth level to be
upgraded while eliminating the hard-coded-key anti-pattern.
