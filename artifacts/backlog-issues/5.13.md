# [Phase 5] Migrate Key Vault to RBAC authorization model

## Description

The Key Vault (`cus1-resume-prod-v1-kv`) uses the **legacy Vault access policy** authorization model (`enableRbacAuthorization: false`). Microsoft recommends migrating to Azure RBAC for consistent access management. This was identified as finding F16 during Phase 0 gap analysis (Issue #15).

Migration steps:
1. Identify all current access policies and map them to RBAC roles
2. Create RBAC role assignments in Bicep (e.g., `Key Vault Secrets User` for Function App managed identity)
3. Switch `enableRbacAuthorization: true` in Bicep
4. Remove legacy access policies
5. Verify Function App can still read secrets via Key Vault references

## Acceptance Criteria

- [ ] Key Vault `enableRbacAuthorization` set to `true` in Bicep
- [ ] RBAC role assignments created for all service principals that need access
- [ ] Legacy access policies removed
- [ ] Function App verified: Key Vault references still resolve correctly
- [ ] GitHub Actions SP has `Key Vault Secrets Officer` or equivalent role

## Dependencies

None — but should be tested carefully in dev before production.

## Copilot Instructions Reference

Follow Azure Verified Modules patterns for Key Vault RBAC. Per copilot-instructions.md: prefer managed identity for Azure service authentication.

## Notes

Source: Phase 0 gap analysis finding F16 (Issue #15). This is a potentially breaking change — all access policies must be migrated to RBAC role assignments before switching. Test in dev environment first.
